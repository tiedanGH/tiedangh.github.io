# Long Night「编辑模式：框选并整体移动墙壁/地形」可行性评估

## 结论
**可实现，且与当前架构兼容，建议分阶段落地。**

当前项目已经具备以下关键基础能力：
- 统一网格坐标（`i,j`）和 `Map` 结构管理单元格。
- 方格（square）与墙（wall）分层编辑能力。
- 完整撤销/重做历史快照。
- 存档导入导出逻辑（可作为移动后状态校验与回滚手段）。

因此，从“可行性”看不是架构级阻塞，而是**交互设计 + 数据搬运规则**问题。

---

## 为什么可行（结合现有实现）

1. 地图单元的逻辑坐标是稳定且可计算的，移动后可直接按偏移写回。
2. 现有代码已经区分并支持：
   - square 的地形/附着/标记
   - wall 的朝向与类型
3. 撤销/重做是整图快照机制，天然适配“批量移动”这种一次性大操作。
4. 当前已有“区块放置”概念，说明批处理改写单元格不是新范式。

---

## 关键技术点与风险

### 1) 选择区域的表达
建议使用“矩形框选 + 可选包含墙体”的最小可用版本：
- 选择范围基于逻辑坐标边界：`minI/maxI/minJ/maxJ`
- 仅允许以偶数位移（保持 square/wall 拓扑）

风险：
- 若允许任意形状（套索/多选），复杂度明显上升。

### 2) 数据提取（Snapshot）
移动前先提取 selection 内部所有单元格状态：
- square：背景图、背景色、附着层、标记
- wall：类型、自定义色、朝向

风险：
- 不能只复制 `backgroundImage`，需要连同 marker/attachment 一并迁移。

### 3) 目标落点与冲突策略
这是实现难点，建议先定义清楚：
- **覆盖策略（推荐首版）**：目标区域直接覆盖。
- 空洞策略：源区域被搬走后恢复为默认未知（或可配置为空地/未知）。
- 越界策略：当前是无限地图，只需 ensure 目标单元存在即可。

风险：
- 墙体和方格边界在目标处可能与已有结构冲突；应统一按“覆盖优先”避免不确定行为。

### 4) 拖动预览（Ghost）
体验上最好有半透明预览，但首版可先做“点击目标点确认移动”。

风险：
- 实时预览会增加渲染与事件处理复杂度。

### 5) 撤销/重做整合
移动操作结束后调用一次历史保存即可。

风险：
- 若在拖动过程频繁写入历史，会污染撤销栈。

## 推荐实现方案（MVP）

### 阶段 A：可用版本（1~2 天）
1. 新增“编辑模式”开关按钮。
2. 鼠标拖拽框选矩形（只选 square + 与其邻接墙）。
3. 记录选区快照。
4. 用户点击目标锚点，按偏移整体搬移。
5. 源区域清空为默认态。
6. 保存一次 history。

### 阶段 B：体验增强（1~2 天）
1. 增加移动前预览。
2. 增加冲突提示（目标区域已有非默认内容时确认）。
3. 增加「复制而非剪切」模式。

### 阶段 C：高级能力（可选）
1. 非矩形多选。
2. 旋转/镜像（需同步处理墙体朝向）。

## 伪代码级流程

```text
enterMoveMode()
  selection = dragSelectRect()
  payload = captureCells(selection)

onTargetChosen(targetAnchor)
  delta = targetAnchor - selection.anchor

  // 1) 先清源（避免同区重叠时数据污染）
  clearSelection(selection)

  // 2) 再写目标
  for each cellState in payload:
    writeCell(cellState.coord + delta, cellState)

  history.saveState()
  exitMoveMode()
```

## 复杂度评估

- **技术可行性**：高
- **实现复杂度**：中
- **主要不确定性**：交互细节与冲突规则定义
- **回归风险**：中（集中在 marker/attachment/wall 三类状态一致性）

---

## 建议先做的验证用例

1. 框选 3x3，包含门墙和普通墙，整体右移 4 格后形状保持一致。
2. 含自定义颜色地形/墙壁的移动后样式完整保留。
3. 含玩家标记与数字标记的区域移动后，唯一标记约束仍成立。
4. 移动后执行撤销/重做，状态完全一致。
5. 跨可视区域（目标点未渲染）移动，结果正确。

---

## 最终建议
优先落地矩形选择的 MVP（覆盖策略 + 一次性提交历史），先验证核心价值；预览和高级选择形状可后续迭代。
